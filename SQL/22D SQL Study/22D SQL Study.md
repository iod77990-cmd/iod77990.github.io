#### 뷰 테이블
- 특징)
  - 데이터베이스에서 테이블의 일부 
또는 ‘특정 조건에 따라 필터링된 데이터만’을 포함하는 ‘가상 테이블’

  -  실제로 데이터를 저장하지 않고 
기존 테이블의 데이터를 기반으로 쿼리를 실행하여 필요한 정보를 반환합니다

  - 데이터 일부분에 집중하거나 복잡한 쿼리를 간소화할 수 있고 
보안, 효율성, 유지보수 측면에서 유용하게 사용될 수 있음

- 장점)
  - 유연성
  - 효율성
  - 보안
  - 가독성
  - 재사용성
  - 통합성

- 단점)
  - 성능저하
  - 업데이트 제약
  - 데이터 일관성
  - 제약조건 : 기본 테이블의 제약 조건을 상속하지 않기 때문에 별도로 제약 조건 걸어야 함

- 고려사항) 
뷰 테이블을 사용하는 경우, 기본 테이블의 데이터 변경에 따라 뷰 테이블도 업데이트 해야 함
이것으로 성능상의 부담을 초래할 수 있고 데이터의 일관성을 유지하기 위해 추가적인 작업이 필요합니다.

- 생성
```sql
CREATE VIEW view_name AS ' ' SELECT column1, column2, ... FROM table_name WHERE condition;
```
- 수정
```sql
CREATE OR REPLACE VIEW view_name AS ' ' SELECT column1, column2, ... FROM table_name WHERE condition;
```

- 이름변경
```sql
DROP VIEW [ IF EXISTS ] view_name;
```

- 조회
```sql
SELECT * FROM information_schema.views WHERE table_schema = 'public';
```


#### 트랜잭션
- 공부하는 분야에서 트랜잭션은 더이상 **분할이 불가능한 업무처리의 단위** 를 의미합니다.

- 특징) 
  - 원자성 : 트랜잭션이 DB에 모두 반영이 되거나 아니면 전혀 반영되지 않아야 하는 특징
  - 일관성 : 트랜잭션이 진행된느 동안에 DB가 변경이 되더라도 처음 트랜잭션을 진행하기 위해 참조한 DB로 진행이 되는 특징
  - 독립성 : 어떤 하나의 트랜잭션이라도, 다른 트랜잭션의 연산에 끼어들 수 없는 특징
  - 영구성 : 트랜잭션이 성공적으로 완료가 되면 결과는 영구적으로 반영되어야 한다는 특징

- 상태)

![트랜잭션의 상태](22D SQL Study)

활성 : 트랜잭션이 정상적으로 실행 중인 상태
부분완료 : 트랜잭션이 마지막까지 실행되었지만 Commit연산이 실행되기 직전의 상태
완료 : 트랜잭션이 성공 이후로 Commit 연산이 실행한 후의 상태
실패 : 트랜잭션 실행에 오류가 발생하여 중단된 상태
철회 : 트랜잭션이 비정상적으로 종료되어 Rollback 연산을 수행한 상태
